# LightConductor — Roadmap

Дорожная карта улучшений по итогам аудита репозитория. Разделена на этапы: от
критических архитектурных задач до QoL-шлифовки. Каждый этап автономен —
можно останавливаться между фазами и выпускать стабильный релиз.

Условные обозначения:

- **[A]** — архитектура / рефакторинг
- **[Q]** — quality of life / UX
- **[I]** — инфраструктура (логирование, тесты, CI, конфиги)

---

## Фаза 0. Фундамент (1–2 недели)

Цель: снять риски потери данных и выстроить инфраструктуру, без которой
следующие фазы будут плодить технический долг.

- **[I] Логирование.** Заменить `print()` и `except Exception: print(e)` в
  `ProjectScreen/ProjectScreen.py:175-192`, `ProjectScreen/PlateLogic/SlaveBox.py`
  и остальных местах на `logging` (ротация в `logs/lightconductor.log`).
  Ошибки, видимые пользователю, показывать через `QMessageBox`.
- **[I] Конфиг.** Вынести hardcoded значения (`masterIp="192.168.0.129"`
  `ProjectScreen.py:205`, `TagLogic/TagScreen.py:31`, UDP-порт `43690`, размер
  чанка `768`) в `config.py` + `settings.json` рядом с `projects.json`.
- **[I] Схема проекта.** Описать JSON-схему `data.json`, добавить поле
  `schema_version`, написать мигратор. Валидацию прогонять при загрузке.
- **[I] Бэкапы.** При сохранении проекта писать `.bak` рядом с `data.json`,
  хранить 3 последних снапшота — защита от краша при автосохранении.
- **[I] Тесты на сериализацию.** Round-trip `pack/unpack` Master→Slave→TagType→Tag
  (сейчас покрыты только use-cases).

Результат фазы: падения логируются, данные не теряются, IP и порты меняются
без правки кода.

## Фаза 1. Домен и application-слой (2–3 недели)

Цель: завершить начатую миграцию в `src/lightconductor/`, убрать прямые
импорты инфраструктуры из виджетов.

- **[A] Единый мэппер сериализации.** Ручная упаковка в
  `ProjectScreen/ProjectManager.py:25-47` и распаковка в
  `ProjectScreen/ProjectScreen.py:131-169` — вынести в
  `src/lightconductor/infrastructure/json_mapper.py`. UI работает только с
  domain-объектами.
- **[A] Убрать LegacyProjectsRepository/LegacyProjectStorage.** Оставить один
  `ProjectRepository` и один `ProjectSessionStorage`, запитанные через DI в
  `main.py`.
- **[A] PatternService.** Перенести `solid_fill`, `floating_gradient`,
  `moving_window`, `sequential_fill` из `SlaveBox.py:200-250` в
  `src/lightconductor/application/patterns/`. Сервис принимает домен и
  возвращает список `(led_index, color, t)` — без Qt-зависимостей.
- **[A] ValidationService.** Пересечения LED-диапазонов, дыры, дубликаты
  пинов, некорректные IP — проверки в одном месте, дергаются перед
  сохранением и перед UDP-аплоадом.
- **[I] Тесты PatternService и ValidationService.** По 5–10 кейсов на каждый.

Результат фазы: виджеты больше не содержат бизнес-логики, `src/` самодостаточен.

## Фаза 2. Разделение SlaveBox и state-management (2 недели)

Цель: разобрать god-object и навести порядок в состоянии.

- **[A] Разбить `SlaveBox` (550 строк, 41 метод)** на:
  - `WaveRenderer` — только pyqtgraph + QMediaPlayer;
  - `TagTimelineController` — interaction layer (клики, hover, drag);
  - `SlaveBoxView` — композиция и лэйаут.
- **[A] Единый store.** `ProjectState` (observable) как источник правды.
  Виджеты подписываются через сигналы; `self.masters = {}` и копии в
  `TagObject`/`TagManager` уходят.
- **[A] Command pattern.** `AddTagCommand`, `MoveTagCommand`,
  `DeleteTagCommand`, `EditRangeCommand` — фундамент под undo/redo и
  автосохранение.
- **[I] Интеграционные тесты** на Command-стек (execute → undo → redo →
  execute повторно).

Результат фазы: любое действие пользователя — это Command, который можно
откатить и сериализовать.

## Фаза 3. Критичные QoL (2 недели)

Цель: то, чего не хватает в ежедневной работе.

- **[Q] Undo / Redo** (`Ctrl+Z` / `Ctrl+Shift+Z`) поверх Command pattern.
- **[Q] Autosave** каждые 30 с + при сворачивании окна. Индикатор «saved»/
  «unsaved» в заголовке.
- **[Q] Горячие клавиши:**
  - `Space` — play / pause;
  - `Ctrl+T` — добавить тег на текущую позицию;
  - `Delete` — удалить выбранный тег;
  - `Ctrl+C` / `Ctrl+V` — копировать/вставить тег;
  - `←` / `→` — шаг по таймлайну (± 100 мс), с `Shift` — ± 1 с.
- **[Q] Drag & drop тегов** по таймлайну с live-обновлением времени и snap
  к сетке.
- **[Q] Zoom / pan таймлайна** колесом мыши; ruler с таймкодом; мини-карта
  трека над основным таймлайном.

Результат фазы: рабочая сессия редактирования перестаёт быть болезненной.

## Фаза 4. Редактор и воспроизведение (2 недели)

- **[Q] Live-превью паттерна** при выборе цвета/типа ещё до apply.
- **[Q] Имитация LED-полосы в UI** — воспроизведение шоу без железа.
- **[Q] Snap to beat.** librosa уже даёт темп — прилипать к долям при
  создании и перетаскивании тега.
- **[Q] Маркеры/цветовые пресеты.** Палитры часто используемых цветов по
  проекту.
- **[Q] Выделение нескольких тегов** (Shift/Ctrl-click, рамкой) + групповые
  операции (сдвиг, удаление, смена цвета).

- **[I] Интеграционный тест** c mock-UDP сервером.

## Метрики успеха

- Размер самого большого класса ≤ 250 строк (сейчас `SlaveBox` ~550).
- 0 прямых импортов инфраструктуры из виджетов (`grep ProjectManager ProjectScreen/` → пусто).
- Coverage `src/lightconductor` ≥ 70 %.
- Autosave + undo/redo закрывают тикеты «потерял работу / случайно удалил».
- Time-to-first-tag для нового пользователя ≤ 2 минуты (измерять на
  playtest-сессии).
